These are the methods that wallets should implement to handle Tron transactions and messages via WalletConnect.

<Note type="warning">
**Please note:** The TRON RPC standard is still under review and specifications may change. Implementation details and method signatures are subject to updates.
</Note>

## WalletConnect Tron Data Structure

### Background

Tron WalletConnect currently defines request and response formats during the connection and transaction signing phases. However, the transaction payloads include a nested two-layer transaction structure, which is unintuitive and introduces unnecessary complexity for both DApps and wallets.

To simplify the data model while preserving backward compatibility, we propose introducing a new session property to distinguish between the legacy and improved structures.

### Problem Statement

#### Current State

- Transaction requests and responses contain a nested `transaction.transaction` structure.
- DApps and wallets must implement additional parsing logic, reducing readability and maintainability.

#### Issues

- The double-wrapped format is not reasonable and adds friction.
- Legacy clients depend on the existing structure, preventing an immediate removal.

### Proposal

During the initial handshake connection, introduce a new property in sessionProperties:

```json
"tron_method_version": "v1"
```

### Behavior Rules

1. **If tron_method_version is not set**
   - Default to the legacy two-layer transaction structure.

2. **If tron_method_version is set to "v1"**
   - Use the simplified single-layer transaction structure.

### Compatibility

#### Backward Compatibility

- Older wallets and DApps that do not recognize `tron_method_version` will continue to operate with the legacy two-layer structure.

#### Forward Compatibility

- New wallets and DApps that detect `tron_method_version = "v1"` can switch to the simplified format without ambiguity.

#### Migration Strategy

- DApps are encouraged to include `"tron_method_version": "v1"` in session properties and migrate once wallet support becomes available.

### Examples

#### Legacy Format (no tron_method_version)

```json
{
  "method": "tron_signTransaction",
  "params": {
    "transaction": {
      "transaction": {
        "raw_data": { "..." },
        "signature": [ "..." ]
      }
    }
  }
}
```

#### Simplified Format (tron_method_version = "v1")

```json
{
  "method": "tron_signTransaction",
  "params": {
    "transaction": {
      "raw_data": { "..." },
      "signature": [ "..." ]
    }
  }
}
```

### 2. New Version

#### Connect Wallet:

In the new wallet connection, the wallet needs add a `tron_method_version` in the `sessionProperties` in the response, which is as follows:

```json
"sessionProperties": {
  "tron_method_version": "v1"
}
```
